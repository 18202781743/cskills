# BUG 280627「解决」流程分析

## 来源

- Bug 页面：`https://pms.uniontech.com/bug-view-280627.html`
- 实际按钮：`解决`
- 按钮链接：`https://pms.uniontech.com/bug-resolve-280627.html?onlybody=yes`

本记录基于实际 PMS 页面与表单源码分析得到，用于固化该站点当前的 `resolve` 流程。

## 实际触发流程

1. 打开 bug 详情页 `bug-view-280627.html`
2. 点击 `解决` 按钮
3. 页面以弹窗 iframe 方式加载 `bug-resolve-280627.html?onlybody=yes`
4. 在弹窗表单中填写字段后点击 `保存`
5. 表单以 `POST` 提交当前 resolve 页面（`<form method='post' ...>`，未显式写 action）

## 前端声明的必填字段

表单源码中的 `window.config.requiredFields` 为：

```text
comment,repair,exttype,symbol,age,source,resolution
```

也就是说，至少要提供：

| 字段名 | 含义 |
| --- | --- |
| `resolution` | 集成状态 / 解决结论 |
| `repair` | 修复点 |
| `exttype` | 类型 |
| `symbol` | 限定符 |
| `age` | 引入阶段 |
| `source` | 来源 |
| `comment` | 备注 |

## 关键字段与默认值

### 1. `resolution`

页面显示名：`集成状态`

实际可选值：

| value | 文案 |
| --- | --- |
| `totask` | 已集成 |
| `checked` | 已验证 |
| `tempcheck` | 临时方案已验证 |
| `tempfix` | 临时方案已解决 |
| `fixed` | 已解决 |
| `custom_checkd` | 定制方案已验证 |
| `retested` | 已转测 |

默认值：空

### 2. `resolvedBuild`

页面显示名：`解决版本`

- 字段名：`resolvedBuild`
- 默认值：空
- 可选项很多，包含 `trunk` 和大量构建版本

### 3. `resolvedDate`

- 字段名：`resolvedDate`
- 默认值：页面打开时自动填当前时间
- 示例：`2026-04-30 12:38:44`

### 4. `repair`

页面显示名：`修复点`

可选值：

- `需求`
- `设计`
- `代码`
- `构建`
- `帮助文档`
- `国际化语言`
- `不涉及`

默认值：空

### 5. `exttype`

页面显示名：`类型`

可选值示例：

- `分配初始化`
- `检查`
- `算法/方法`
- `功能/类/对象`
- `定时/序列化错误`
- `接口错误`
- `依赖错误`
- `不涉及`
- `需求和规格说明不完整`
- `需求变更`
- `实现细节分析不足`
- `技术可行性分析不足`
- `不实际的客户期望`

默认值：空

### 6. `symbol`

页面显示名：`限定符`

可选值：

- `丢失`
- `错误`
- `外来`（页面文案显示为“无关”）
- `不涉及`

默认值：空

### 7. `age`

页面显示名：`引入阶段`

可选值：

- `历史版本引入`
- `当前版本新增引入`
- `当前版本修复引入`
- `不涉及（终端产线禁用）`

默认值：空

### 8. `source`

页面显示名：`来源`

可选值：

- `内部开发`
- `lib库调用`
- `外包代码`
- `代码移植`
- `不涉及`

默认值：空

### 9. `assignedTo`

页面显示名：`指派给`

- 字段名：`assignedTo`
- 当前页面默认值：`ut003782`
- 页面文案：`Ut003782:王若璇`

### 10. `comment`

页面显示名：`备注`

该字段在页面中已预置模板，默认内容如下：

```text
【版本号】：必填项，修复的组件包名称-版本。
【自测环境镜像版本】：必填项，修复的镜像版本，xxx.iso。
【代码地址】：必填项，修复代码提交的地址。
【根因分析（Bug）】：必填项，Bug 产生的根因，提炼研发经验。
【影响范围】：必填项，代码层面分析对其他代码模块的影响范围，是否要同步修改。
【自测结果截图/视频】：必填项，描述自测结果。
【自测架构】: 必填项
【引入责任人】：选填项，根据提交记录填写对应的责任人，无提交记录填写历史遗留。
【引入提交链接】：选填项，根据提交记录填写代码提交链接，无提交记录填写历史遗留。
【维护线是否回合主线】：选填项，维护线修复后填写合入主线的结果以及原因，主线修复无需填写。
【测试点】：选填项
【前置条件】：选填项
【步骤】：选填项
【预期】：选填项
【自测结果】：选填项，二选一 （通过/不通过）
```

## 其他字段

| 字段名 | 说明 |
| --- | --- |
| `duplicateBug` | 当 `resolution` 需要重复类语义时使用 |
| `buildProject` | 解决版本所属项目 |
| `createBuild` | 是否新建版本 |
| `ishardWare` | 是否硬件相关 |
| `analysis` | 引入分析（由 `age` 联动展示） |
| `files[]` | 附件上传 |
| `uid` | 编辑器/上传相关隐藏字段，由页面脚本注入 |

## 对脚本的含义

当前 skill 若要真正稳定触发该 PMS 的 `resolve` 流程，至少应支持：

1. 固定跳转 URL：`bug-resolve-{bug_id}.html?onlybody=yes`
2. 必填字段校验：`resolution / repair / exttype / symbol / age / source / comment`
3. 对 `resolution` 使用受控选项而非自由输入
4. `comment` 默认值优先复用页面模板，再允许用户覆盖
